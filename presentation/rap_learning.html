RAPs by example

An introduction to Reproducible Analytical Pipelines (RAPs)

Sean Manzi

Dartington Service Design Lab

What are RAPs?

  • A partially or fully automated statistical and analytical processes
  • Most effectively used for analysis and reporting that is routinely undertaken
  • They operate from data extraction through to report production
  • They incorporate software engineering best practices to produce reproducible, auditable, efficient and high quality outputs

Why use RAPs?

RAPs will:

  • Improve the quality of an analysis
  • Create a more efficient process reducing the replication of effort
  • Increase stakeholder trust in the analysis process and its outputs
  • Improve the continuity of analytical outputs

What must a RAP do?

  • Minimise the manual steps in the process
  • Include a detailed and well documented process for it’s implementation
  • Be built using open source software (e.g. R or Python)
  • Be persistently accessible and auditable using version control such as Git
  • Be open to anyone using file and code sharing platforms such as GitHub
  • Contain well commented code and high quality documentation

Some considerations

  • There may be some restrictions on the ability to automate parts of the pipeline due to the databases and software being used by an organisation
  • Not everything will be able to be shared due to sensitive content. This is most often present in the inclusion of ‘real’ data to run alongside the code. In this instance, best practice is to provide a ‘dummy’ dataset containing fake data that allows anyone to understand the shape and type of data being used by the RAP and run the code
  • RAPs are most effectively built collaboratively with more than one developer and input into the process from the relevant stakeholders (e.g. data providers and output users)

RAP file descriptions

  • report1.csv - raw data extract in comma delimited format
  • data_cleaning.R - R script to perform cleaning operations on the data and output a new data file
  • data_transform.R - R script to perform data transformations required for the analysis and output a new data file
  • analysis_and_vis.R - R script used for development and testing of the data analysis and visualisation production for the report
  • rap_report.qmd - Quarto report script which incorporates the analysis_and_vis.R code to produce a dynamic report
  • rap_run.sh - A bash shell script used to run the cleaning and transformation R scripts then render the quarto report in the desired output format

Data extract guidelines

  • Data is best extracted in a CSV (comma delimited) format as it does not retain any formatting that might cause issues in the pipeline
  • Rows in the data should represent individual records
  • Columns should represent the variables for the records
  • Column names are used often in the analysis, using all lower case and separating words with an underscore (’_’) are the common conventions for ease and to reduce errors
  • Any data table should only have one row of column headers
  • Dates should be of a consistent format across the data table otherwise this needs to be rectified in the cleaning phase
  • Unique identifiers are used to identify the records of individuals within and across data tables
  • Data tables should not have any formatting such as colours, boarders, different font sizes etc. Use of the CSV format helps to ensure this

Data cleaning guidelines

Data cleaning is the process of consolidation, ensuring consistency across the data and fixing any known errors.

It normally involves tasks such as:

  • Ensuring consistent categorisation within columns
    • Removing leading or trailing whitespace
    • Consolidating similar categories with inconsistent naming
  • Reformatting dates to ensure consistency across the data table
  • Ensuring missing data is correctly and consistently identified across the data table
  • Removing any unnecessary and incorrect data

Data transformation guidelines

This is the stage where the data is reshaped, merged or new data created from existing variables

It normally involves tasks such as:

  • Removing unnecessary columns from the data
  • Merging one or more data tables either
    • horizontally - adding rows (e.g. data tables with the same columns)
    • vertically - adding columns (e.g. data tables with different columns against the same unique identifiers)
  • Creating new columns by performing transformations on one or more existing columns (e.g. calculating the length of time between two dates)

Analysis and visualisation development

  • During the report development process, it is a good idea to develop and test all of your code for analysis and visualisations outside of the report.

  • Develop this code in an R script before transferring it to the report file.

  • This will be an iterative process. As you write the report, you will likely find additional variables, tables and figures that you want to produce. Write and test them in the development script before transferring the code to the report.

  • You will want your analysis to be focused on the needs of those who will use the report. Their requests for information will change as they see what is possible with the data. Again, expect the report development process to be iterative. Working with reproducible code and version control makes this process of iterative development auditable and transparent.

Report development guidelines

The report in this example is written in Quarto. This is a flexible markdown language which allows you to incorporate text and code with dynamically produced variables and figures.

  • When doing development work on your report, it is best to render it in html format for ease of viewing and speed
  • Make the text as generic as possible so that the numbers can be changed dynamically
  • Avoid hard-coding values which are likely to change from one use to the next
  • Use if statements to insert dynamic text which is dependent on the values produced in the analysis to ensure the written interpretation is relevant to the analysis output

Automation using a shell script

Shell scripts are used to pass commands directly to the computer outside of any one specific programme. In Mac OS and Linux OS’s these are .sh files and are run from the terminal. On Windows OS these are CMD files run from the command line. Shell/command files perform a series of commands in order

  • Make sure the script includes a reference to the file path or paths where your files are located
  • Quarto documents can be rendered in html, pdf or word formats

General guidance

  • Always begin a script by setting your working directory. This means that the script will always know where to look for files and folders referred to in the script no matter where it is being run from

Using Git and GitHub for version control

Git repositories provide hosting, version control and management for your code base

You will want to setup your own GitHub account and then create a GitHub organisation for you and your colleagues to collaborate on

You will need to install Git (Windows) and if using windows it might be useful to install GitHub Desktop if you are not used to working from the command line
